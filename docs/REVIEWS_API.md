# Отзывы на лендинге: ТЗ для бэкенда (my.pitchcv.app)

Интеграция блока отзывов на **pitchcv.app** с API и модерацией на **my.pitchcv.app** (там же приложение и админка).

---

## Финальное сообщение команде бэкенда — обсуждение закрыто, можно начинать разработку

Ниже зафиксированы договорённости после ревью текущего кода (**`api.py`**, `get_admin_user` / `ADMIN_EMAIL`) и продукта. Отклонения только по согласованию с владельцем продукта.

1. **Роли `admin` / `editor`:** в **v1** трактуем **editor = admin** — доступ к модерации отзывов у того же списка пользователей, что и к админке (`get_admin_user`, один **`ADMIN_EMAIL`** или расширение env одним списком **`ADMIN_EMAILS`** через запятую). Отдельная роль **`editor` в БД / токенах** и эндпоинт вида **`get_reviews_moderator`** — **отдельная задача на бэклог**, иначе формальное «две роли» без доработки auth не выполнить.

2. **Honeypot — зафиксированный ответ в бэкенд-контракте:** при срабатывании honeypot (например поле заполнено в **multipart**, если появится) — **`204 No Content`**, тело пустое, **запись не создаём**. Так бот не отличает «поймали» от тихого успеха сильнее, чем при явном **400**. *(Для обычной успешной отправки остаётся **201** + `{ id, status }`.)*

3. **`would_recommend`:** с лендинга в **JSON** — строки **`"yes"` | `"no"`**; в **БД** хранить **boolean**. В **`GET /api/reviews/stats`** описать **`recommend_percent`** **только** из доли «да» среди опубликованных с **ненулевым** ответом:  
   `recommend_percent = round(100 * count(would_recommend == true) / count(all_published_with_non_null_recommend), 1)`  
   **Не смешивать** с правилом «только звёзды 4–5», если выбрана эта формула.

4. **JSON-LD и публичный API — жёсткое правило:** поля **`aggregateRating`** и каждый элемент **`review[]`** в разметке должны строиться **из тех же строк**, что отдают **`GET /api/reviews/stats`** и **`GET /api/reviews/public`** (или один агрегирующий endpoint). **Нельзя** добавлять в schema отзывы, которых **нет** в видимом HTML страницы.

5. **Публичные карточки и бейджи:** в объекте **`GET /api/reviews/public`** явно отдавать **`verified: boolean`** и **`source`**: `"native"` | `"imported"` | … — чтобы лендинг мог честно показать «Verified» / «Imported» **без хардкода**.

6. **`PATCH /api/reviews/:id` (админ):** поле **`pinned`** (bool). **Порядок в публичной выдаче:** сначала **закреплённые** (`pinned == true`), затем обычная сортировка (`recent` / `rating` и т.д.). Правки **`title` / `body`** — только орфография/очевидные опечатки, без смысла; **версионирование в v1 не обязательно**, но желательны **`updated_at`** и **кто менял** (хотя бы **`last_moderator_id`**).

7. **`published_at`:** при **первом** переводе в **`approved`** выставить **`published_at = now()`** один раз. **Повторные** правки текста (орфография) **`published_at` не перетирать** — для SEO и честности дат.

8. **Rate limit:** одного лимита **по IP** мало за **NAT**. Зафиксировать **комбинацию**, например: **до 10 созданий отзыва с одного IP за час** и **до 3 с одного нормализованного email за сутки** (точные числа можно подкрутить, главное — зафиксировать в конфиге и доке).

## 2. CORS

- Разрешить для лендинга **`Origin: https://pitchcv.app`** (и при необходимости `www`, превью-стенды, `http://localhost:*` для разработки).
- Методы: минимум **GET**, **POST**, **OPTIONS**; заголовки **Content-Type**, **Accept** — по фактическим запросам.
- **Админские** маршруты модерации — только с **my.pitchcv.app** (cookie / session / Bearer), отдельный CORS с маркетингового домена не обязателен.

## 3. Публичные эндпоинты (для лендинга)

### 3.1. `POST /api/reviews`

Приём отзыва из формы «Leave a review». Сразу **не публикуется**: статус **`pending`**.

**Тело (JSON), согласованное с формой на лендинге:**

| Поле | Обязательно | Примечание |
|------|-------------|------------|
| `author_name` | да | публичное имя |
| `author_email` | да | не показывать в публичном API |
| `rating` | да | целое 1–5 |
| `would_recommend` | да | `"yes"` \| `"no"` в JSON с лендинга; в БД хранить **boolean** |
| `title` | да | заголовок отзыва |
| `body` | да | текст |
| `author_role` | нет | job title / role |
| `country` | нет | |
| `feature_tag` | нет | например `ats-check`, `cv-editor`, `cover-letter`, `keyword-scanner` |
| `consent_to_publish` | да | `true` (boolean в JSON) |
| `consent_to_process` | да | `true` (boolean в JSON) — **единственное каноническое имя**; не использовать `consent_to_data_processing` |

**Антиспам:**

- **Honeypot:** поле **`fax_extension`** в форме на лендинге; в **`application/json`** с лендинга **не передаётся**; основная серверная защита — **rate limit** (+ опционально captcha). Если бэкенд в будущем принимает multipart и видит заполненный honeypot — ответ по контракту: **`204 No Content`**, без записи (см. финальное сообщение вверху).
- **Rate limit:** **комбинация IP + нормализованный email** (за NAT одного IP мало). Ориентир: **10/час на IP**, **3/сутки на email** — зафиксировать в env и не менять «тихо».
- Captcha — по необходимости позже.

**Ответ:** успех — **201** + `{ "id": "...", "status": "pending" }`; лимит — **429** с понятным телом. Honeypot (если применим на сервере) — **204** без тела.

---

### 3.2. `GET /api/reviews/public`

Выдача только опубликованных отзывов: **`status = approved`** и **`consent_to_publish = true`**.

**Query (минимум):** `limit`, `offset` (или cursor), опционально `sort` (`recent` | `rating`).

**Порядок по умолчанию:** сначала **`pinned == true`** (закреплённые модератором), затем **`sort`**: например `recent` → `published_at` DESC; tie-break согласовать с фронтом.

**Элемент ответа (публичные поля):** `id`, `author_name`, `rating`, `title`, `body`, `published_at`, **`verified`** (boolean), **`source`** (`"native"` \| `"imported"` \| …), опционально `feature_tag`. Поле **`pinned`** в публичном JSON **не обязательно** (достаточно сортировки на бэке). Email и `admin_notes` не отдаются.

---

### 3.3. `GET /api/reviews/stats`

Агрегаты **только по тем же отзывам**, что попадают в публичную выдачу (`approved`, `consent_to_publish`).

- `average_rating` — среднее по `rating`, округление согласовать с фронтом (например **1** знак после запятой).
- `review_count` — число таких отзывов.
- **`recommend_percent`** — **только** из булева **`would_recommend`** среди опубликованных отзывов с **заполненным** полем (исключить `NULL` из знаменателя):  
  `recommend_percent = round(100 * count(yes) / count(all_published_with_non_null_recommend), 1)`  
  Правило «доля 4–5★» **не смешивать** с этой формулой.

Цифры в шапке лендинга и в **JSON-LD** обязаны совпадать с этим ответом и с списком из **§3.2** (см. **§3.4**).

### 3.4. JSON-LD и публичный API (жёсткое правило)

`aggregateRating` и каждый объект в `review[]` в `<script type="application/ld+json">` должны быть **вычислены из тех же данных**, что отдают **`GET /api/reviews/stats`** и **`GET /api/reviews/public`** (или один агрегирующий endpoint с тем же содержимым). **Запрещено** включать в schema отзывы, которых **нет** в видимом HTML страницы. Лендинг **не дорисовывает** отзывы в schema сверх API.

## 4. Админка и модерация (my.pitchcv.app)

**Доступ (v1):** сейчас в коде админка завязана на **`get_admin_user`** / **`ADMIN_EMAIL`**. Для отзывов в **первой версии**: **editor = admin** — те же пользователи; при необходимости расширить один **`ADMIN_EMAIL`** до списка **`ADMIN_EMAILS`** в env. Отдельная роль **editor** в БД/токенах и отдельный guard (например **`get_reviews_moderator`**) — **отдельная задача на бэклог** (см. блок «Финальное сообщение» вверху).

**Статусы отзыва:** `pending`, `approved`, `rejected`, `hidden` (скрыт с сайта, запись в БД сохраняется).

**`published_at`:** выставляется **один раз** при **первом** переходе в **`approved`** (`published_at = now()`). Последующие правки орфографии **`title`/`body`** и другие PATCH **не сбрасывают** `published_at`.

**Минимальный набор API (все под `/api`, только для авторизованных админов):**

| Метод | Назначение |
|--------|------------|
| `GET /api/reviews` | Список с фильтрами: `status`, диапазон дат, `rating`, пагинация |
| `PATCH /api/reviews/:id` | `status`, **`verified`**, **`pinned`** (bool), правки **`title` / `body`** только орфография/опечатки без изменения смысла, **`admin_notes`**. Желательно обновлять **`updated_at`** и **`last_moderator_id`** (или эквивалент). Версионирование текста в **v1 не обязательно**. |
| `GET /api/reviews/export.csv` | Экспорт |

Публичный порядок выдачи: **сначала `pinned`**, затем параметр сортировки — см. **§3.2**.

Опционально: полный журнал модерации (кто/когда).

**Продуктовые правила:** не отклонять только из‑за низкой оценки; негативные и 3–4★ допустимы. Отклонять спам, оскорбления, явно чужие персональные данные, оффтоп.

---

## 5. Модель данных (поля БД, ориентир)

`id`, `author_name`, `author_email`, `author_role`, `country`, `rating` (1–5), **`would_recommend` (boolean в БД; вход POST — строки yes/no)**, `title`, `body`, `feature_tag`, `source` (`native` | `imported` | …), `verified`, **`pinned`**, `consent_to_publish`, `status`, **`published_at`**, `created_at`, `updated_at`, опционально **`last_moderator_id`**, `admin_notes`, `helpful_count`, `language`.

Импорт (CSV / ручная запись в админке) — отдельный сценарий с явным `source` и при необходимости `consent_received` в БД.

## 7. Что передать фронту лендинга (pitchcv.app)

1. **База API:** `REVIEWS_API_BASE=https://my.pitchcv.app/api` (без завершающего `/`). На сайте: `<meta name="reviews-api-base" content="https://my.pitchcv.app/api">` и/или до подключения `main.js`: `window.REVIEWS_API_BASE = 'https://my.pitchcv.app/api'` (альтернатива: `window.__REVIEWS_API_BASE__` — поддерживается в текущем лендинге).

2. **Поля `POST /reviews` (JSON):** строго как в **п. 3.1**. Согласие на обработку данных — только **`consent_to_process`** (boolean). Имя **`consent_to_data_processing`** в контракт **не использовать**.

3. **Honeypot:** в HTML имя **`fax_extension`**; в **JSON не включать**. Скрытие — CSS (например clip/размер 1×1), **без** опоры только на `display:none`; **`tabindex="-1"`**, **`autocomplete="off"`**; контейнер **`aria-hidden="true"`**. См. ответ бэка при срабатывании honeypot — **§3.1** / блок «Финальное сообщение» (**204**).

4. **JSON-LD:** обязательное правило — **§3.4**. На статике: **SSR**, **build/revalidate** из **`GET /reviews/stats`** + **`GET /reviews/public`**, либо временно ручная синхронизация (не в долгую).
